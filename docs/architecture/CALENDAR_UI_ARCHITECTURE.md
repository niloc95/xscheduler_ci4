# Calendar UI Architecture

Covers the visual design system, layout structure, and color contracts for the scheduler interface. For module responsibilities and JS architecture, see `docs/architecture/scheduler_ui_architecture.md`.

---

## Shell Layout

The scheduler renders inside a `.scheduler-shell` container that fills 100vh. The shell is divided into a fixed toolbar and a 62%/38% content split.

```
.scheduler-shell          flex-column, 100vh
  └─ .scheduler-toolbar   54px, var(--color-surface-0), border-bottom
  └─ .content-split       flex-row, remaining height
       ├─ .cal-pane        62% — scrollable calendar view (Today/Day/Week/Month)
       └─ .right-panel     38% — provider cards + slot grid
```

Mobile (≤ 1023px): columns stack vertically; right panel limited to 40vh.

Source: `resources/scss/pages/_appointments-scheduler.scss`

---

## View Layouts

### Day View

Per-provider vertical columns, side by side inside `.cal-pane`.

- One column per visible provider
- 100px per hour vertical height
- Fixed time column on the left edge
- Sticky provider headers at the top of each column
- Now-line spans all provider columns, updates every 60 seconds
- Appointment cards scale content to available pixel height (5 tiers — see `scheduler_ui_architecture.md`)

### Week View

7-day chip grid inside `.cal-pane`. Not a time-slot grid.

- Each day = one cell; shows 2–4 appointment chips, then "+N more"
- Selecting a day refreshes the right panel accordion, which groups appointments by provider
- No vertical time axis in the week grid itself

### Month View

42-cell grid (6 rows × 7 columns) inside `.cal-pane`.

- 2 visible chips per cell, "+N more" expand button
- Provider availability indicator bar and provider dots at the cell bottom
- Daily detail section renders outside `.cal-pane` into `#daily-provider-appointments`
- Day cell click navigates to the Day view

### Today View

Compact list view, not a time-slot grid.

- Hero stats row: Total / Upcoming / Completed / Free Slots
- Upcoming appointments (full opacity) and completed (65% opacity) listed below
- No time columns; appointments ordered chronologically

## Color System

All color logic is centralized in `resources/js/modules/scheduler/appointment-colors.js`.

### Status Colors

Five statuses, each with a light and dark mode palette:

| Status | Light bg | Light border | Dark bg | Dark border |
|---|---|---|---|---|
| `pending` | `#FEF3C7` amber-100 | `#F59E0B` amber-500 | `amber-900/80` | `amber-500/50` |
| `confirmed` | `#DBEAFE` blue-100 | `#3B82F6` blue-500 | `blue-900/80` | `blue-500/50` |
| `completed` | `#D1FAE5` emerald-100 | `#10B981` emerald-500 | `emerald-900/80` | `emerald-500/50` |
| `cancelled` | `#FEE2E2` red-100 | `#EF4444` red-500 | `red-900/80` | `red-500/50` |
| `no-show` | `#F3F4F6` gray-100 | `#6B7280` gray-500 | `gray-700/80` | `gray-400/50` |

**Light mode:** solid opaque backgrounds with dark text for readability.  
**Dark mode:** semi-transparent dark backgrounds with light text.

Each status entry exposes: `bg`, `border`, `text`, `dot`, `chipBg`, `chipBorder`, `badgeBg`.

`getStatusColors(status, darkMode)` is the public accessor. Normalizes `no_show` / `noshow` → `no-show`.

### Provider Colors

Provider colors are stored in `xs_users.color` (hex string). `getProviderColor(provider)` returns `provider.color` or falls back to `DEFAULT_PROVIDER_COLOR = '#3B82F6'` (blue-500) from `constants.js`.

`getProviderTheme(provider)` returns a full theme object (bg, border, text, dot) derived from the stored color. Eight provider color themes are pre-defined.

Provider initials are generated by `getProviderInitials(name)` which delegates to `getAvatarInitials()` from `utils/avatar.js`.

### Surface Tokens

Layout and toolbar use CSS custom properties from the design system:

| Token | Purpose |
|---|---|
| `var(--color-surface-0)` | Toolbar and pane background (light) |
| `var(--color-border)` | Toolbar bottom border |
| `var(--color-gray-800)` | Dark mode toolbar |
| `var(--color-gray-700)` | Dark mode toolbar border |

## Dark Mode

The theme source is `document.documentElement.dataset.theme`. Set to `'dark'` for dark mode, absent or `'light'` for light mode.

Color tokens switch via `[data-theme="dark"]` CSS selectors in the SCSS. JS components detect dark mode via `document.documentElement.dataset.theme === 'dark'` and pass `darkMode=true` to `getStatusColors()`.

See `resources/js/dark-mode.js` and `resources/js/utils/dark-mode-detector.js`.

## Accessibility and Responsiveness

- All appointment chips and cards have sufficient color contrast (dark text on light bg in light mode; light text on dark bg in dark mode)
- Status is conveyed via both color and text label — color alone is never the sole indicator
- Keyboard navigation is preserved; scroll traps avoided via `overscroll-behavior: contain` where applicable
- Mobile breakpoint: columns stack at ≤ 1023px per Rule #4 in `Agent_Context_v2.md §Rule #4` (Dashboard Layout and Scroll UX Contract)

---

## Styling Constraints

- No inline `style=` attributes in scheduler view templates (CSP compliance)
- Dynamic position values (top, height, left percentages) use `data-style` JSON attributes applied via post-render script
- All layout uses Tailwind utility classes or the SCSS design tokens above
- New status colors must be added to both `STATUS_COLORS` and `STATUS_COLORS_DARK` in `appointment-colors.js`
